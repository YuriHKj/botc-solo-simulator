# UI 重构执行文档：魔典主视图优先（供后续改码参考）

## 1. 目的

本文件用于指导下一轮 UI 重构，把当前“多面板仪表盘”改为“中心魔典主视图”架构，确保对局内视觉焦点稳定在魔典圈盘，其他信息以抽屉/弹窗/浮层方式出现。

适用范围：
- `index.html`
- `styles.css`
- `scripts/ui.js`
-（少量）`scripts/app.js`

## 2. 当前问题基线（已确认）

### 2.1 布局结构问题

1. 页面虽是单列壳体，但中央区被硬拆为三段：
   - 魔典区
   - 对话舞台
   - 底部卡组（新手引导+恶魔伪装）
2. 即使“收起底部”，中段对话舞台仍占固定高度，导致魔典无法扩张到主视图。
3. 左右栏是绝对定位浮层，信息密度高时仍与中心视觉竞争。

### 2.2 样式维护问题

1. `styles.css` 存在多轮主题叠加（旧三栏网格 + 新主题 + patch 覆盖），规则重叠较多。
2. 一些旧规则已失效但仍保留，增加后续改动风险。

### 2.3 体验问题

1. 玩家进入对局后第一感知不是“魔典”，而是“HUD + 大块舞台/面板”。
2. 关键流程（私聊、公聊、提名）信息容器分散，视觉路径不连续。
3. 低分辨率/窗口化时，魔典圈盘容易显得偏小。

## 3. 目标体验（重构后）

### 3.1 核心原则

1. 对局内默认态：魔典占据最大视觉权重（主视区 75%~85%）。
2. 常驻信息最小化：保留顶部轻 HUD + 中心魔典。
3. 行动/对话信息按需出现：
   - 私聊：弹窗舞台
   - 日志：右抽屉
   - 引导：可折叠浮卡
4. 所有非魔典内容默认“弱存在”，而非“常驻大面板”。

### 3.2 量化标准

1. 1920x1080 下，魔典圈盘直径目标 >= 可视区域短边的 62%。
2. 默认态下，底部区域不占固定高度（仅悬浮小条或折叠）。
3. 对局内首屏（无弹窗）时，用户第一注视应在魔典中心区域。

## 4. 目标信息架构

### 4.1 页面层级（新）

1. 顶层：`app-shell`（两行）
   - `hud-panel`
   - `play-stage`
2. `play-stage` 内：
   - `board-wrap`（主层）
   - `floating-dock`（可折叠小组件：引导、伪装、阶段提示）
   - `drawer-left` / `drawer-right`（抽屉，不默认抢占高度）
3. 模态层保持现有：开始菜单、剧本手册、设置、夜间行动、私聊大弹窗。

### 4.2 状态模式

统一增加 UI 模式类（挂在 `app-shell`）：
- `mode-focus`：魔典主视图（默认）
- `mode-dialogue`：私聊/公聊展示增强
- `mode-night-action`：夜间选择优先
- `mode-review`：结算复盘（显示 AI 摘要）

## 5. 分阶段改造计划

## Phase A：清理与控基线（先做）

### A1. 样式分层重排

1. 在 `styles.css` 明确分段注释：
   - `Base`
   - `Layout`
   - `Components`
   - `Overlays`
   - `Responsive`
2. 将旧三栏时代规则移到 `legacy` 段并统一禁用（或删除）。
3. 避免同一选择器在文件中出现 3 次以上覆盖。

### A2. 加“布局健康检查”

在开发期加入临时调试类（如 `.debug-layout`），快速检查：
- 可视区边界
- 中心圈盘占比
- 抽屉是否遮挡 token

## Phase B：中心魔典主视图重建（核心）

### B1. 中央区从“三段固定高度”改为“魔典优先”

目标：
- `center-column` 改成以 `board-wrap` 为主，其他内容改悬浮或抽屉。

做法：
1. `center-column` 行模板从
   - `1fr + debate + bottom`
   改为
   - `1fr`（主舞台）
2. `debate-stage` 改为：
   - 默认折叠的小型信息条（底部）
   - 仅在 `mode-dialogue` 展开。
3. `center-bottom` 改为 `floating-dock`：
   - 默认收纳
   - 点击 HUD 按钮弹出。

### B2. 魔典尺寸算法升级

当前 `fitBoardToSquare` 采用固定扣边距（`-80/-110`）。
改为“比例安全边距”：
- `safeWidth = bounds.width * 0.92`
- `safeHeight = bounds.height * 0.90`
- `size = min(safeWidth, safeHeight)`

再叠加上下限：
- `min = 420`
- `max = min(bounds.width, bounds.height) * 0.98`

## Phase C：侧栏改抽屉（减少视觉竞争）

### C1. 左抽屉（操作）

1. 默认关闭，只保留一枚明显入口按钮。
2. 打开后宽度分两档：
   - `compact`（仅阶段按钮）
   - `full`（全部表单）
3. 非必要字段收进“高级选项”。

### C2. 右抽屉（日志/复盘）

1. 默认仅显示简短“最近事件胶囊”。
2. 完整日志放右抽屉。
3. AI 推理摘要仅在 `mode-review` 且抽屉打开时显示。

## Phase D：对话与流程整合

### D1. 私聊

1. 以 `chatDramaModal` 为主入口（已具备雏形）。
2. 左栏的私聊表单降级为备用入口。
3. 私聊结束后可继续追问，不强制回左栏。

### D2. 公聊

1. `debate-stage` 使用“轮次时间轴”但默认弱化显示。
2. `mode-dialogue` 下展开并高亮当前发言者 token。

## Phase E：响应式与全屏体验

### E1. 分辨率策略

1. 16:9 桌面：魔典优先，抽屉悬浮。
2. 16:10 或窄屏：抽屉覆盖式，魔典仍居中。
3. 小屏（<1024）：自动切 `compact HUD`，保留魔典交互。

### E2. 全屏策略

1. 首次进入对局自动建议全屏（不强制）。
2. 窗口化时动态降低外围装饰透明度，避免视觉噪音。

## 6. 文件级实施指引

## `index.html`

1. 重命名/重组中心区域：
   - `center-column` -> `play-stage`
   - `debate-stage` 与 `center-bottom` 迁为浮动组件。
2. 抽屉容器统一：
   - `leftPanel` / `rightPanel` 语义明确为 drawer。
3. HUD 仅保留高频按钮：
   - 主菜单、设置、剧本手册、抽屉开关。

## `styles.css`

1. 先删冗余重复块，再定义单一来源样式。
2. 以状态类驱动布局：
   - `.mode-focus`
   - `.mode-dialogue`
   - `.drawer-left-open`
   - `.drawer-right-open`
3. 调整 `z-index` 规范，避免“面板遮挡弹窗/魔典焦点”。

## `scripts/ui.js`

1. 新增 UI 状态机：
   - `layoutMode`
   - `leftDrawerOpen`
   - `rightDrawerOpen`
   - `dockExpanded`
2. `renderGame` 中按状态切换模式类（而不是散落条件）。
3. `fitBoardToSquare` 改为比例算法，且在抽屉开合后重新计算。
4. 把“按钮可用性”和“布局可见性”逻辑解耦。

## `scripts/app.js`

1. 阶段切换时通知 UI 模式：
   - 私聊/公聊进入 `mode-dialogue`
   - 夜间行动进入 `mode-night-action`
   - 结算进入 `mode-review`
2. 保持规则引擎不变，只改前端呈现状态。

## 7. 风险与规避

1. 风险：重构 CSS 造成现有控件消失/不可点。  
   规避：每阶段后做交互回归（新局、私聊、公聊、提名、夜间）。
2. 风险：抽屉动画影响性能。  
   规避：仅动画 `opacity/transform`，避免 `width/height` 高频重排。
3. 风险：移动端断点下 token 被遮。  
   规避：断点下强制抽屉覆盖式，不占中心区。

## 8. 验收清单（完成定义）

1. 默认进入对局时，魔典圈盘明显大于当前版本。
2. 不打开抽屉时，左右不出现大面板文本块。
3. 私聊可在弹窗连续进行，不依赖左栏。
4. 右侧 AI 摘要只在复盘态显示。
5. 1920x1080、1600x900、1280x720 三档无严重遮挡。
6. 全流程可玩：开局 -> 私聊 -> 公聊 -> 提名 -> 夜晚 -> 结算。

## 9. 建议执行顺序（最小风险）

1. 先做 Phase A（清理）
2. 再做 Phase B（魔典主视图）
3. 然后 Phase C（抽屉）
4. 再做 Phase D（对话整合）
5. 最后 Phase E（响应式/全屏细修）

执行原则：每个 Phase 都可独立运行并打包验证，不跨阶段混改。
