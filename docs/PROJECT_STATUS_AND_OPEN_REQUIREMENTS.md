更新时间：2026-04-30

## 1. 结论摘要

当前项目已经从“单页 Demo”推进到一个可运行、可打包、带三剧本和 AI 对话雏形的 Electron 桌面原型。它可以继续作为私有开发主线推进。

但如果按“官方级规则 + 沉浸式单机 BOTC 游戏”的目标衡量，仍有几类核心需求没有完全满足：

1. **BMR / SnV 不是完全官方级**：多数角色已有触发与日志，但很多需要 Storyteller 裁量或特殊输入的能力仍是概率、随机或简化处理。
2. **角色交互 UI 仍偏通用**：很多角色需要选角色、输入问题、输入多组猜测、选择蓄力/不行动/多杀，目前主要还是“选玩家”。
3. **对话系统可用但还不够像真人**：已有意图识别、快捷问法、信念模型和轻量训练接入，但自由文本理解、同日口径一致性、公聊辩论节奏仍有限。
4. **UI 目标部分达成但仍可精修**：魔典主视图、抽屉、弹窗已有重构，但文档中的“魔典占据 75%-85% 主视觉、底部不常驻抢空间”等目标需要继续实机验证和细调。
5. **规则状态边界还需要继续收紧**：AI、玩家主观信息、Storyteller 全知信息、中毒/醉酒客观状态之间的隔离已经有基础，但还没有彻底结构化为统一事件/知识模型。
6. **验证体系不足**：当前以手动/内联烟测为主，缺少固定种子回归测试、角色级单元测试和 UI 自动截图断言。

## 2. 当前已满足的主要需求

### 2.1 基础可玩性

- 三剧本可开局：TB / BMR / SnV。
- 5-15 人配置可按官方人数结构抽取。
- 支持随机初设与测试用自选身份。
- 支持第一夜、白天、后续夜晚与阶段推进。
- 支持私聊、公聊、提名、投票、处决、跳过处决。
- 支持通用善恶胜负判定与部分特殊胜负。
- 支持打包 Windows Electron `.exe`。

### 2.2 TB 规则

- TB 已拆入 `scripts/roles/tb.js`，整体是当前完成度最高的剧本。
- 已覆盖 Baron 外来者修正、Drunk 伪装、Poisoner、Imp 传位、Scarlet Woman、Saint、Mayor、Virgin、Slayer、Butler、Monk、Ravenkeeper 等核心逻辑。
- Investigator 中毒/醉酒误信息已经朝“结构化假信息”方向改进，而不是完全随机。
- TB 主动夜间技能有 Storyteller 风格交互元数据。
- Slayer 白天技能已接入日间行动接口。

### 2.3 BMR / SnV 核心覆盖

- `scripts/roles/bmr.js` 和 `scripts/roles/snv.js` 已存在模块化 runner。
- 两剧本多数角色已有核心触发、日志、死亡/保护/醉毒/特殊胜负分支。
- `each night` 与 `each night*` 已按夜间顺序表分流，已验证 Fortune Teller / Imp / Pukka / No Dashii / Pit-Hag 等关键窗口。
- Evil Twin、Klutz、Sweetheart、Barber、Mastermind、Vortox 等关键分支已有实现雏形。

### 2.4 UI

- 局内以魔典圆桌为中心。
- 支持角色 token、未知 token、死亡帷幕、提醒物、角色标记、恶魔伪装。
- 支持剧本手册/夜间顺序查看。
- 支持开始菜单、设置入口、主菜单入口。
- 支持夜间行动弹窗和私聊弹窗。
- 支持侧栏/底部折叠，减少常驻信息压迫。

### 2.5 AI / 对话

- AI 不是纯随机：会读取声明、投票、死亡、夜间信息、邪恶互认等信号。
- 私聊有意图识别：怀疑、理由、信任、报身份、投票、夜间信息、比较、计划等。
- 快捷按钮已作为主路径，自由输入作为补充。
- 公聊能按嫌疑和证据生成发言。
- 邪恶方首夜互认后，同阵营信息互通已有基础。
- 语料管线和轻量模型已落地到 `scripts/ml_runtime_model_data.js`。

### 2.6 文档与仓库

- 私有仓库已初始化并推送。
- README 已补充非官方声明、运行方式、打包方式、发布风险。
- 角色实现审计、UI 重构计划、AI 对话报告、模块拆分记录均已存在。

## 3. 未完全满足的需求

### P0：影响“完整可玩一局”的缺口

#### 3.1 BMR / SnV 的非玩家目标输入

未满足点：

- Gambler 应选择“玩家 + 角色”，当前猜测来源仍偏自动/公开声明。
- Po 应可选择“不杀蓄力”或蓄力后三杀，目前人类 UI 与三杀不完全对齐。
- Philosopher 应选择“角色能力”，当前近似为选玩家/随机镇民并直接改身份。
- Artist 应输入是/否问题，当前简化为“目标是否邪恶”。
- Juggler 应输入最多 5 组“玩家 + 角色”猜测，当前可自动生成。
- Pit-Hag 应选择目标变成的角色，当前随机变形。
- Cerenovus 应选择强迫声明角色，当前随机角色。
- Barber 死亡后应由恶魔选择两名玩家交换，当前随机。

建议：

- 先做通用 `action schema`，支持 `chooseRole`、`choosePlayerRolePairs`、`yesNoQuestion`、`multiTarget`、`chargeOrKill`。
- 再逐角色接入，而不是为每个角色单写 UI。

#### 3.2 死亡触发与 Storyteller 弹窗

未满足点：

- Ravenkeeper、Moonchild、Klutz、Barber、Sage 等死亡触发角色应在触发时弹出选择/信息窗口。
- 当前不少触发仍由系统随机选择或直接写日志。

建议：

- 增加 `pendingStorytellerAction` 队列。
- 夜晚/白天死亡处理后，如果产生人类可操作触发，则暂停结算并弹窗。
- UI 确认后继续 resolve。

#### 3.3 完整流程引导仍可增强

未满足点：

- 玩家能玩，但“下一步该做什么”仍依赖文本提示和按钮状态。
- 新手仍可能不知道先私聊、公聊、提名、跳过/投票、夜间行动之间的关系。

建议：

- 做一个 `currentObjective` 面板或 HUD 胶囊，只显示下一步动作。
- 引导不放大段文字，改成阶段目标 + 一个主按钮。

### P1：规则准确性缺口

#### 3.4 BMR 官方级偏差

重点偏差：

- Lunatic 与真实恶魔目标同步/误导未完全实现。
- Godfather 仅覆盖外来者被处决后的额外击杀，官方触发细节更丰富。
- Mastermind 延迟结算仍是简化。
- Zombuul 诈死状态 UI 与规则表达不够官方化。
- Pukka 毒杀顺序近似。
- Shabaloth 复活是固定概率，不是 ST 裁量。
- Po 人类三杀/蓄力 UI 不完整。

#### 3.5 SnV 官方级偏差

重点偏差：

- Snake Charmer 互换后的阵营/邪恶信息叙事需要补强。
- Mathematician 统计口径不完全等同官方。
- Savant 信息模板少，不能根据局势生成高质量一真一假。
- Philosopher 选择能力语义不对。
- Artist / Juggler 明显简化。
- Mutant / Cerenovus 的疯狂判断依赖声明字段或概率，不是真正理解发言。
- Pit-Hag 不能选择新角色。
- Vigormortis 死亡爪牙继续行动的 UI 和信息呈现较粗。
- Vortox 未保证所有善良信息严格错误。

#### 3.6 主观信息 / 客观状态 / AI 知识边界

已完成基础：

- 中毒/醉酒客观状态不直接暴露给普通主视角。
- 私有信息可进入事件日志，并带 `private` / `playerId`。

仍未满足：

- AI 检索上下文未完全接入 `private-info` 的权限边界。
- 邪恶方共享信息、主观夜间信息、公开发言、ST 全知状态还没有统一成严格知识模型。
- AI 可能仍在某些场景使用过多全局状态或过少私有信息。

建议：

- 建立 `knowledge_view(playerId, state)`，所有 AI 决策只能通过它读信息。
- 把 `events.infoPings`、`privateNotes`、`logs`、公开声明统一映射为可授权证据。

### P1：UI / 体验缺口

#### 3.7 魔典视觉权重还需继续确认

目标文档要求：

- 默认态魔典占主视区 75%-85%。
- 底部区域不应固定占高度。
- 右侧 AI 摘要只在复盘态显示。
- 1920x1080、1600x900、1280x720 三档无严重遮挡。

当前状态：

- 已做多轮重构，但仍需实机截图验证当前版本是否达到目标。
- 对话舞台、侧栏、底部浮层历史上多次与魔典争空间，需要持续防回归。

建议：

- 固化 UI 回归截图脚本，至少保存三种分辨率截图。
- 把 `UI_GRIMOIRE_FOCUS_REFACTOR_PLAN.md` 的验收清单转成自动/半自动检查。

#### 3.8 人物立绘与对话沉浸

已做：

- 有占位立绘/对话弹窗方向。

未满足：

- 角色或 AI 人物立绘未形成稳定资产体系。
- 私聊/公聊还未完全变成“角色登场 + 轮流辩论”的演出。

建议：

- 先用自制占位立绘系统，不依赖官方素材。
- 每个 AI 分配 persona、立绘占位、语气 profile。

### P1：AI 对话缺口

#### 3.9 自由文本理解仍有限

未满足：

- 多意图问句不能稳定拆解。
- 否定、反问、复杂比较仍可能误判。
- AI 同日口径锁未完全实现。
- 公聊是批量/回放式，缺少可中断的逐句辩论节奏。

建议：

- 保留按钮化主路径。
- 给自由输入增加二段式解析：`intent candidates + target candidates + confidence`。
- 低置信时让 AI 回答“我理解你是在问 X，如果不是可以换个问法”，而不是硬猜。
- 给每个 AI 增加 `dayStanceMemory`，锁定当天核心怀疑/保护对象。

#### 3.10 训练模型接入仍偏辅助

已完成：

- 真实语料接入与轻量模型训练流程存在。

未满足：

- 模型更多作为表达/投票倾向辅助，没有成为完整 NLU 或策略核心。
- BOTC 专属人工小样本不足。

建议：

- 不做 LLM 训练，继续走“规则状态机 + 小模型分类 + 检索模板”路线。
- 补 50-200 条 BOTC 专属标注小样本：中毒误报、醉酒自信、邪恶互认、死人发言、疯狂发言。

- 角色规则回归主要靠内联脚本和人工验证。

#### 3.12 文档存在陈旧和冲突

表现：

- `CURRENT_THREAD_HANDOFF.md` 仍写“继续修复 mojibake”，但最近一次已做大范围清理。

建议：

#### 3.13 大体积研究缓存影响阅读

风险：

- 长线程中读取这些文件容易拖慢上下文。

建议：

- 普通开发不读 `docs/research/` 大文件。
- 若未来公开仓库，必须移除抓取素材与官方缓存。
- 私有仓库可保留，但 README/INDEX 已提醒不要批量加载。

## 4. 建议下一阶段路线

### 路线 A：先补交互接口，再补角色

适合目标：让 BMR/SnV 从“近似可玩”走向“官方级可操作”。

1. 建立通用 action schema。
2. 改夜间/死亡触发弹窗支持非玩家目标。
3. 优先接入 BMR：Gambler、Po、Moonchild、Professor。
4. 再接入 SnV：Artist、Juggler、Pit-Hag、Barber、Cerenovus。
5. 更新角色审计和固定种子测试。

### 路线 B：先稳 UI 体验

适合目标：让游戏看起来更像一个完整作品。

1. 按 `UI_GRIMOIRE_FOCUS_REFACTOR_PLAN.md` 做截图回归。
2. 把私聊入口完全收敛到魔典点人 + 弹窗。
3. 公聊做逐句时间线和 token 高亮。
4. 底部/侧栏默认弱化，魔典保持最大。

### 路线 C：先稳 AI

适合目标：让电脑玩家更像人。

1. 建 `knowledge_view(playerId, state)`。
2. 接入 private-info 到 AI 可见证据。
3. 加 day stance memory，减少自相矛盾。
4. 补 BOTC 专属小样本，训练/检索只作为辅助。

## 5. 推荐优先级

我建议下一步走“路线 A + 少量 B”：

1. 先做通用 action schema，这是 BMR/SnV 官方级细则和沉浸式 Storyteller 弹窗的共同底座。
2. 同时把私聊入口继续收敛到魔典点人弹窗，减少 UI 重复。
3. 每完成 2-3 个角色，补一个固定种子回归测试。

理由：  
UI 和 AI 的很多问题都源自“角色事件不够结构化”。先把角色行动接口补好，后续 AI 才能准确引用“我问了什么/我猜了什么/我被强迫声明什么”，UI 也能真正像说书人一样逐步引导。
